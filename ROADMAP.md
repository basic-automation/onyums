- [ ] Add concurrency/backpressure limits — code spawns a task per rendezvous circuit and per stream without explicit limits. Add configurable semaphores: max circuits, max streams per circuit, max total streams, max body size before WAF inspection, per-handler timeouts. — *scope note:* the *per-circuit* caps (max streams per circuit, request rate, byte budget) already exist in `onyums_skin::AccountingCircuitPolicy` (`max_streams`/`max_request_rate`/`max_bytes`) and are wired via `.circuit_policy(...)`. What remains is *host-global* backpressure.
  - ***Max-concurrent-circuits slice landed 2026-07-20 — and the "not offline-verifiable" reason this item carried was retired by the `serve_loop` seam landed the same run.*** The old scope note said a host-global semaphore "is a live-Tor-path change (a semaphore in the hot loop is not offline-verifiable and can deadlock if mis-wired), so land it with the live tier". That was true when the loop could only be driven by real `RendRequest`s. With `IncomingCircuit` + `serve_circuits_with` in place it is directly testable, so this landed offline with tests — a concrete payoff for the seam, and a reminder that "needs the live tier" is sometimes a statement about the current *test seams*, not about the change.
  - *What it is:* `OnionServiceBuilder::max_circuits(n)` caps concurrently-served rendezvous circuits. A permit is taken **after** the circuit policy admits the circuit and held for the circuit's whole lifetime, so it is returned however the circuit ends. At capacity the loop **refuses** rather than queues (`try_acquire_owned`, which never waits) — the same stance `ConnectionLimit` takes on raw ports, and for the same reason: queueing converts a circuit flood into unbounded memory growth plus a latency cliff for clients already in service, and a refused client can retry where one parked in an invisible queue cannot tell slow from dead. Unset stays unbounded (today's behaviour); `max_circuits(0)` is a clean `serve()` error offline rather than a silently unreachable service.
  - *Refusals get their own counter*, `ServiceMetrics::circuits_refused_at_capacity` (+ `onyums_circuits_refused_at_capacity_total` in the Prometheus exposition, now seven families). Deliberately **not** folded into `circuits_rejected`: a policy rejection is a verdict about that circuit, while a capacity refusal says nothing about the circuit and everything about the service being full. Conflating them would make an under-provisioned service look like one under attack — exactly backwards for the operator deciding whether to raise the limit or shed traffic upstream.
  - *The tests pin the property that matters and are shaped so the bad outcome is visible:* the admitted circuits park in their driver and never finish, so their permits are still held when the next circuit is offered (with a driver that returns immediately, no cap of any size would ever be reached and the test would prove nothing). Two permits, four circuits → two accepted, two refused, `circuits_rejected` still `0`. **If the loop queued instead of refusing, that test would hang rather than fail** — which is the failure a cap exists to prevent, so the assertion shape is part of the coverage.
  - ***Max-concurrent-streams slice landed 2026-07-20, same run.*** `OnionServiceBuilder::max_streams(n)` caps concurrently-served streams **across all circuits**. It is not a duplicate of the circuit cap: a circuit cap bounds how many clients are in service, while this bounds the total work in flight — one circuit may open many streams, so a circuit cap alone does not bound sockets or memory. At capacity the stream is refused with Tor's **`RESOURCELIMIT`** end reason (which says truthfully that the service is full, rather than `DONE`, which would claim the request was simply finished) and the circuit plus its other streams live on — the same shape as a policy rejection. Counted as `ServiceMetrics::streams_refused_at_capacity` (+ `onyums_streams_refused_at_capacity_total`; the exposition is now eight families); `max_streams(0)` is a clean offline `serve()` error.
    - *One ordering subtlety worth keeping:* the disposition is recorded **after** the permit is taken, not before. Recording first would count a capacity-refused stream as *both* served and refused, making the served total a lie — and a test pins exactly that (`streams_served == 2` while `streams_refused_at_capacity == 3`).
    - As with the circuit cap, the tests park every served stream so its permit is still held at the next offer, and a queueing regression would **hang** rather than fail.
  - ***Per-handler-timeout slice landed 2026-07-20, same run — and it is what makes the two caps mean anything.*** `OnionServiceBuilder::handler_timeout(Duration)` bounds how long one served stream may run. Without it a client holds its permit for as long as it likes simply by going quiet mid-request, so `max_circuits`/`max_streams` bound *how many slow clients it takes to fill the service* rather than bounding the damage — a timeout is what turns a permit into a **lease**. A stream that exceeds it is dropped and counted as `ServiceMetrics::streams_timed_out` (+ `onyums_streams_timed_out_total`; nine families now), which is deliberately its own counter: a rising timeout count against flat refusals is the signature of *slow* clients rather than *too many*, and the two call for different responses.
    - **Off by default, and that is a design decision rather than caution.** A long-lived stream is a legitimate use of an onion service — the README's own websocket example is one — so a default here would silently sever working applications.
    - *Test note worth keeping for the next person:* the test uses tokio's **paused clock** (`#[tokio::test(start_paused = true)]`, which needs tokio's `test-util` feature — not part of `full`, so it was added as a dev-dependency), asserting a 30s timeout fires without spending 30s or racing a real one. Two traps, both hit while writing it: under `start_paused` the clock only auto-advances when the runtime is **idle**, so a `yield_now` loop keeps it busy and it never moves — advance explicitly; and the advance must come **after** the spawned task has registered its timer, or it lands before the timeout exists and does nothing.
  - *Still open on this item:* a max body size before WAF inspection (which is `onyums-skin`'s side of the boundary, not the server's).

No `remove_dir_all` on a runtime worker.

- [x] Explore [rama](https://github.com/plabayo/rama) — **evaluated 2026-07-16. Recommendation: do not adopt it as onyums' HTTP stack; there is nothing to flush out.** rama 0.3.0 (published 2026-07-07, MIT/Apache-2.0 — <https://crates.io/crates/rama>) is a modular *service* framework for building network stacks explicitly: proxies (reverse/HTTP(S)/SOCKS5/SNI/MITM), HTTP services, clients with pooling, traffic inspection, and lower-level TCP/UDP/DNS.   - *Why not:* adopting it means rewriting the HTTP stack to gain capabilities aimed at problems onyums does not have.   - *Not a reason, struck 2026-07-16:* I originally called rama's **1.96 MSRV** a "hard blocker". It is not one — onyums is a frontier project and a frontier MSRV is fine (owner's call), so an MSRV delta must never decide a technical question here. The recommendation stands on the scope mismatch alone, which is the honest reason. What *does* still count against it: carrying a second service-trait vocabulary next to tower's, in a crate whose entire surface is "hand us your axum Router".
  - *The three genuinely interesting bits, recorded so this stays evaluated rather than re-litigated:* (1) its **TLS-fingerprinting / traffic-inspection** work is the closest thing to skin's JA4H fingerprinting — worth reading for ideas, though skin's is Tor-native by design and rama's keys on things Tor strips; (2) its **ACME automation** would matter *if* CA-signed `.onion` certs (HARICA, `Tls::Provided`) ever gain an ACME path — today that is manual, so it is a watch item, not work; (3) its **explicit TCP/UDP service model** is the same shape as onyums' `StreamHandler`/`route_port`, i.e. independent confirmation the raw-port design is idiomatic, not an argument to switch.
  - *Watch condition (the only thing that would reopen this):* if rama ships a Tor/onion transport, or if onyums ever grows a real proxy/MITM story, re-evaluate. Neither is on this roadmap.

        type Req: IncomingStreamReq;
        type Streams: Stream<Item = Self::Req> + Send + Unpin;

        async fn reject(self) -> Result<()>;
    }
    trait IncomingStreamReq {                        // what StreamRequest provides

        async fn accept(self) -> Result<OnionStream>;
        async fn reject(self, reason: EndReason) -> Result<()>;

    }
