- [x] Kill the global `ONION_NAME` singleton тАФ per-service handle returned from the builder

- [x] Split src/lib.rs тАФ **done.** It was ~108 KB / ~2983 lines holding the public API types, Tor bootstrap, service lifecycle, status projection, Skin integration, TLS layering, port-router assembly, and builder logic at once. lib.rs is now **~241 lines** and is purely the crate's front door: the crate docs, the module list, and the `pub use` surface (plus the arti re-exports and the one live-Tor test, which drives the whole stack and so belongs to no single module). Every module is self-contained with its own tests and the public surface is unchanged throughout тАФ `cargo test -p onyums --lib` never moved off 167 passing across the final slices.
  - *The modules, and what each owns:* `address.rs` (the `OnionAddress` value type) ┬╖ `status.rs` (`ServiceStatus`/`ServiceProblem`/`ServiceHealth` + the projections of arti's `#[non_exhaustive]` state) ┬╖ `metrics.rs` (`ServiceMetrics`/`CircuitMetrics` + the Prometheus exporter) ┬╖ `connection.rs` (`ConnectionInfo`) ┬╖ `tor_client.rs` (client bootstrap, storage dirs, the ephemeral state-dir lifecycle, keystore hardening) ┬╖ `service_config.rs` (offline `OnionServiceConfig` assembly + restricted discovery) ┬╖ `http_stack.rs` (the gate+HSTS+app composition) ┬╖ `tls_setup.rs` (the `Tls` policy тЖТ `TlsAcceptor` assembly) ┬╖ `port_router.rs` (port тЖТ handler dispatch + raw-port exposures) ┬╖ `serve_loop.rs` (the live rendezvous accept loop) ┬╖ `handle.rs` (`OnionServiceHandle` + `Drop`) ┬╖ `builder.rs` (`OnionServiceBuilder`/`OnionService`/`serve` + the launch path).
  - *The organising principle the split ended up encoding, worth keeping:* the untestable live path is now **one module** (`serve_loop.rs`), and it holds no decisions тАФ every verdict it acts on is made in a module that is offline-tested (`circuit_gate`, `port_router`, `tls_setup`, `http_stack`), and the loop only sequences them. Likewise `builder.rs::serve` is ordered so that everything that can fail offline fails *before* the Tor bootstrap. If a future change puts a decision into `serve_loop.rs`, that is the signal it belongs somewhere testable.
  - *Fixes the split surfaced along the way, rather than pure moves:* a **panic on the launch path** (`generate_simple_self_signed(...).unwrap()` тЖТ a contextful error; rcgen rejecting a SAN is reportable config, not a reason to abort the caller's process), a crate-internal `OnionServiceHandle::new` so the handle's seven fields stay private to the module owning their invariants, two clippy warnings in `project_service_status`, and a pure `subject_alt_names` split out so the name landing in the certificate is testable without parsing X.509 back out of a `rustls` config.

- [x] Preserve upstream error contexts тАФ `setup_tor_client`/`launch_onion_service`/the nickname parse and the three `StreamRequest::accept()` sites now use `map_err(|e| anyhow!("...: {e}"))` (was `|_|`), carrying arti's cause through for operators. Offline-tested on the nickname site (the one reachable without live Tor).
- [x] Add CI тАФ `.github/workflows/ci.yml` on push/PR + a weekly cron, five jobs: **test** (`cargo test --workspace --all-features`, plus an explicit keystore-permissions step since those tests are `#[cfg(unix)]` and inert on a Windows dev box), **clippy** (`--workspace --all-targets --all-features -- -D warnings`), **msrv** (a matrix тАФ onyums on 1.90, onyums-skin on 1.89, checked separately so skin's floor can't drift up to the server's arti-imposed one), **deny** (see the Phase 2 cargo-deny item), and **docs** (`cargo doc --no-deps --workspace --all-features`). Every job's command was verified locally; CI itself has never run (the first PR is its first execution). *Two follow-ups split out below: the `fmt --check` job and tightening docs to `-D warnings`.*
- [ ] Add a `cargo fmt --check` CI job тАФ **blocked on a repo-wide reformat first**: `rustfmt.toml` sets `unstable_features = true`, so formatting needs a *nightly* rustfmt, whose output drifts between releases. Needs: one reformat commit pinned to a specific toolchain (a `rust-toolchain.toml` or a pinned nightly in the job), which is a large, owner-facing diff and a deliberate decision, not a routine side effect.

- [ ] Add live Tor integration tests тАФ current tests are offline-only mocks via tower. The roadmap itself flags raw-TCP serve, multi-service, and status-event emission as needing "real Tor runs." Add an --ignored or CI-gated live test tier. тАФ *slice landed:* the tier exists: `live_service_serves_over_the_tor_network_and_shuts_down` (`#[ignore]`, run via `cargo test -- --ignored`) launches an ephemeral service on the real network, then a second Tor client (`allow_onion_addrs` тАФ the serving client's config rejects `.onion` dialing) does an end-to-end HTTPS fetch through a real rendezvous circuit and must get the app body back, then shuts down cleanly, all under timeouts. Live finding baked into the test: arti 0.43 reports `Running` only after descriptor uploads to BOTH HsDir rings succeed, so `ready()`/`is_reachable` lags de-facto reachability by minutes тАФ the e2e fetch, not status, is the pass signal (see the test's doc comment; feeds the arti-upgrade item below); *next slices:* live raw-TCP serve, multi-service on a shared client, status-event emission.
- [ ] Add concurrency/backpressure limits тАФ code spawns a task per rendezvous circuit and per stream without explicit limits. Add configurable semaphores: max circuits, max streams per circuit, max total streams, max body size before WAF inspection, per-handler timeouts. тАФ *scope note:* the *per-circuit* caps (max streams per circuit, request rate, byte budget) already exist in `onyums_skin::AccountingCircuitPolicy` (`max_streams`/`max_request_rate`/`max_bytes`) and are wired via `.circuit_policy(...)`. - [x] Add cargo audit/cargo deny to CI тАФ `deny.toml` + a `deny` job running the config **twice**: the default graph, and `--workspace --all-features`. The pair is the point: the default run mechanically enforces "the default build stays MIT/copyleft-free" (the allow list has no copyleft and skin's LGPL `equix` is opt-in), while the all-features run is the only place `LGPL-3.0-only` is permitted тАФ and only for the two named crates `equix`/`hashx`, never blanket. `[graph] targets` is pinned rather than inherited, because arti's TLS reaches `native-tls` тЖТ `openssl-sys` on Linux but `schannel` on Windows, so an unpinned run means a green verdict on a Windows dev box says nothing about the Linux runner. Running it found real problems, fixed rather than ignored: **anyhow 1.0.102 тЖТ 1.0.103** (RUSTSEC-2026-0190, UB in `Error::downcast_mut` тАФ onyums threads `anyhow::Error` through every fallible path), **spin 0.9.8 тЖТ 0.9.9** (yanked), and a **publish blocker** (`onyums-skin` was a path-only dependency, which crates.io rejects тАФ see the crates.io cross-cutting item). Four advisories remain ignored, each transitive inside arti and each carrying its reasoning; the one to re-check on every arti bump is **RUSTSEC-2023-0071** (the "Marvin Attack" timing sidechannel in `rsa` 0.9.x, which has *no* patched release), reached only via arti's ssh-key fork for OpenSSH key file formats тАФ onyums handles no RSA key (ed25519 identity, x25519 client auth), which is a believed-unreachable argument and **not a proof**.
- [ ] Reconsider the `bans` allowance for `openssl-sys` тАФ `deny.toml` bans it *except* through arti's own TLS runtime (`onyums тЖТ tor-rtcompat тЖТ async-native-tls тЖТ native-tls тЖТ openssl{,-sys}`), which is an honest statement of where the "100% Rust, no FFI" rule already has an exception on Linux/macOS rather than a purity the tree has. Closing it for real means moving arti off `native-tls` to its rustls runtime (`tor-rtcompat` offers one) and confirming the `static`/vendored-OpenSSL build feature is no longer pulled тАФ a dependency-surface change worth doing deliberately, ideally alongside the arti upgrade.
- [x] Zeroize secret material тАФ `ClientAuthKeypair` now `impl Zeroize`/`ZeroizeOnDrop` + `Drop`, wiping the x25519 secret in place on drop (the public half is left intact). `zeroize = "1.8"` added to `[workspace.dependencies]` (pure Rust, already transitive via the dalek stack тАФ no new version). Offline-tested. *Note:* `secret_bytes()` still returns an owned copy the caller must scrub.
- [x] Enforce filesystem permissions for keystore тАФ `src/keystore_perms.rs` enforces `0700` on directories / `0600` on files across the whole state tree on every bootstrap (persistent *and* ephemeral), before arti opens it, and **fails closed**: a path that cannot be made owner-only aborts the launch rather than serving a locally-readable identity. Design: the policy (`permission_bits`/`is_too_permissive`/`tighten`) is pure integer logic compiled and tested on every platform, and only the syscall layer is `#[cfg(unix)]`; the module is `std`-only so the Unix branch can be compiled and *run* standalone (the 9 syscall tests are inert on the Windows dev box, so they were executed under WSL against a real Unix filesystem тАФ and CI's Linux runner now runs them too). `is_too_permissive` is stricter than "world-readable" (any group bit counts тАФ a group-readable key leaks the identity just as completely); `tighten` narrows rather than overwrites, so a deliberate `0400` stays `0400`; the state dir is born `0700` via `DirBuilder::mode` rather than `create_dir_all`, which applies the umask and would leave it `0755` exactly while the identity key is first written; symlinks in the tree fail closed rather than being chmod'ed through. Complements arti's own `fs-mistrust` (which refuses to start on a lax tree) by *repairing* the common lax-umask/restored-from-tar case first. Off Unix it is a documented no-op (`Hardening::Unsupported`, logged) rather than a false claim тАФ Windows is ACL-based and `std` exposes only a read-only bit. Documented in the README (Deployment тЖТ "Keystore permissions", plus three troubleshooting rows for the failure modes it introduces).
- [x] Document restricted-discovery limits prominently тАФ the warning was one sentence at the *end* of the section, after the example (i.e. where someone who already copied the snippet will not read it). It is now a block directly under the definition, and it says what goes wrong rather than only asserting the label: removing a client does not revoke it (the allowlist gates descriptor *decryption*, so a removed client works until republication and its circuits die тАФ no session kill), the key is a bearer credential indistinguishable from the real client if leaked, and every authorized client is equal (no roles, no per-route permissions, no audit trail). Shows the app-auth layering rather than only advising it, with the composition spelled out (discovery keeps strangers from opening a circuit тЖТ the Skin gate absorbs abuse тЖТ app auth decides what a user may do) plus a test a reader can apply: if leaking one `.auth_private` would be a breach rather than an inconvenience, you are using discovery as authentication.
- [x] Label the WAF as best-effort/non-authoritative тАФ a new README "What the gate does *not* do" states it plainly: a starter *signature* ruleset is pattern matching, signature WAFs are bypassable, and a determined attacker with time to iterate will get past this one. Names what it is for (cutting drive-by noise, raising scanning cost) and what it must not displace (parameterised queries, output encoding, authorising every request), and reframes a WAF block as telemetry rather than proof of safety.
- [ ] Add raw-port security controls тАФ RawTcpHandler bypasses the HTTP Skin/WAF entirely. Add per-port allowlists, connection limits, auth hooks, and explicit warnings when exposing SSH/admin services.
  - *Warnings slice landed:* `PortRouter::exposures() -> Vec<RawPortExposure>` reports what each registered raw port gives up (no Skin gate, no WAF, no rate limiting, no built-in TLS), and `serve()` logs one WARN per port at launch тАФ the decision to open a raw port is made in code that may be far from whoever operates the service. A `SENSITIVE_PORTS` table names the well-known administrative/datastore services (SSH, Telnet, LDAP, SMB, MSSQL, Docker API, etcd, MySQL, RDP, PostgreSQL, AMQP, VNC, Redis, Kafka, Elasticsearch, Memcached, MongoDB) so the line is specific rather than boilerplate; it is deliberately conservative (unambiguous assignments only) because a "sensitive" warning that fires on every port is noise. Pure data, offline-tested (`exposures()` sorts by port тАФ the backing map's order is randomised, so unsorted output would shuffle between runs; tests also pin that no entry collides with the reserved HTTP ports and that the table is sorted/duplicate-free). `RawPortExposure`/`well_known_sensitive_service` are re-exported so an operator can assert on this in their own startup checks. README's "Protocol versatility" now states the bypass instead of only claiming raw ports are "never a relaxation of the secure HTTP defaults" (true of the HTTP path, quietly misleading about the raw port).
  - *Next slices:* the enforcement half тАФ per-port **connection limits** (a semaphore per registered port; note this touches the live accept path, so land it with the live tier), a per-port **auth hook** (a `StreamHandler` wrapper that must approve a stream before the backend sees it), and a per-port **client allowlist** keyed on restricted-discovery identity if arti ever surfaces which client authenticated a circuit (it does not today тАФ see the CircuitId item).
- [x] Fix Under-Attack "Challenge" for raw TCP тАФ `stream_disposition(action, port)` is now port-aware: `Challenge` serves only on the reserved HTTP ports (80/443, where the Skin gate renders the challenge) via the new `port_router::is_reserved_http_port`, and **fails closed to Reject** on any raw port. Offline-tested.
- [ ] Treat CircuitId as short-lived only тАФ it's synthetic (Arti doesn't expose a durable circuit identifier); attackers can rotate circuits. Bind longer-lived controls to clearance tokens, client certs, or restricted-discovery keys. тАФ *doc half landed:* the README's "What the gate does *not* do" now states that the gate keys on the circuit and the clearance token rather than an identity, that a `CircuitId` is synthetic and rotatable at will, and that per-circuit limits are therefore a speed bump rather than a ban. *Still open (the code half):* audit that no onyums control actually treats a `CircuitId` as durable, and give `CircuitPolicy` implementors a documented, token-keyed path for anything that must outlive one circuit.
- [x] Guaranteed ephemeral cleanup тАФ **solved, but not with signal trapping**, because the item as written could not work: it named OOM/SIGKILL as the problem and `tokio::signal` on SIGINT/SIGTERM as the fix, and SIGKILL is by definition uncatchable тАФ no handler runs for the exact case that matters (nor for a power loss, nor a `SIGKILL`ing container runtime). A signal handler would have covered only the graceful exits `shutdown`/`Drop` already cover, while a library that grabs the app's Ctrl-C changes the host program's behaviour uninvited.
  - *What landed instead тАФ an ownership claim + a launch-time sweep* (`src/tor_client.rs`): each ephemeral state dir holds a locked `.onyums-owner.lock` (`EphemeralClaim`), held by `OnionServiceHandle` for the service's lifetime. An advisory file lock is released by the **OS** when the holder exits тАФ *however* it exits тАФ so "can this lock be taken?" is a sound proxy for "is the owner gone?". `sweep_stale_ephemeral_dirs` runs at each ephemeral launch and removes every `onyums-ephemeral-*` dir whose claim is free, cleaning up after whatever died last time. This covers SIGKILL/OOM/power-loss, which no signal approach can.
  - *Why it is safe where a heuristic would not be:* a live sibling's keystore is never touched no matter how long it has run (an "older than an hour" rule would delete the identity of a service that has been up for a week). PID-liveness would also have worked but needs FFI (`kill(pid, 0)`), which the no-FFI rule forbids тАФ `std`'s `File::try_lock` (stable since 1.89, under the 1.90 MSRV) needs none and is cross-platform.
  - *Teardown ordering is encoded in the type:* `EphemeralIdentity::release()` consumes `self`, dropping the claim before handing back the path, because the lockfile lives *inside* the directory and Windows refuses to remove a directory holding an open handle тАФ a removal attempted while the claim was alive would silently fail there and leave the identity key on disk.
  - Offline-tested (7 tests): a claimed dir survives, an abandoned one is swept, a mixed tree is resolved in one pass, unrelated temp dirs and prefix-sharing *files* are untouched, a dir with no lockfile counts as abandoned, the sweep is idempotent, and a claim is exclusive.
- [x] Document "double TLS" semantics тАФ done, but **re-scoped rather than written as specified**, because the item's premise does not hold. It asked to articulate that inner TLS keeps traffic "from local Tor exit logic to Axum" encrypted "against local memory snooping" as "a real security property". It is not one here: arti runs in-process and the TLS terminates in that same process (`src/lib.rs` accepts the TLS stream and serves the axum app inline), so an attacker who can read this process's memory holds the TLS keys *and* the plaintext. An onion service also has no exit node тАФ the circuit is already encrypted end-to-end тАФ so the inner layer is not a second lock against a network eavesdropper either. Documenting the stated rationale would have taught readers a property onyums does not have. The README's new "Why TLS *inside* Tor?" says what actually holds, which is reason enough: TLS is what makes an onion service a first-class web origin (browsers gate WebCrypto, service workers, `Secure`/`__Host-` cookies, and HTTP/2 on the *scheme in the URL bar*, not on how the bytes travelled), the app gets a correct `https` view of the request, and `Tls::Strict` fails closed on a stray plaintext listener. It is equally explicit about the limits, including that raw `route_port(...)` handlers are deliberately not TLS-wrapped.
